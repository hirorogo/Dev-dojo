# SOV38 Helper Toolkit

xperableフォーク ([GitHub](https://github.com/hirorogo/xperable)) に同梱しているPythonラッパーツール。BLアンロック・TAバックアップ・Magisk導入をガイド付きで実行できます。

## インストール

```bash
git clone https://github.com/hirorogo/xperable
cd xperable

# xperable本体をビルド
make

# ヘルパーツールを起動
python3 sov38_helper.py
```

  6) Magisk root化
  7) フルガイド (1→5→6 を順番に実行)

コマンドラインからも直接実行できます：

```bash
python3 sov38_helper.py --check     # 環境チェック
python3 sov38_helper.py --status    # デバイス確認
python3 sov38_helper.py --backup    # TA + boot バックアップ
python3 sov38_helper.py --unlock    # BLアンロック
python3 sov38_helper.py --magisk    # Magisk root化
```

## 自動リトライエンジン

xperableエクスプロイトの核心機能。失敗時に自動でリトライし、さらにUSBバッファサイズを変えて最適値を探索します。

### なぜリトライが必要なのか

xperableのエクスプロイト (CVE-2021-1931) は **USBバッファオーバーフロー** を利用しています。原理を簡単に説明すると：

```
通常のfastbootコマンド:
  PC → [USBパケット: "getvar:version"] → SOV38
  PC ← [USBパケット: "OKAY0.4"]        ← SOV38

xperableのエクスプロイト:
  PC → [巨大なUSBデータ: ペイロード + オーバーフロー部分] → SOV38
                                                        ↓
                                            ABLのバッファ境界を超えて
                                            コード実行領域を上書き
                                                        ↓
                                            任意コード実行を達成！
```

この「巨大なUSBデータ」を送る際、OSのUSBスタックがデータを **自動的にチャンク（分割）** して送信します。

### USBペイロード分割の問題

USBプロトコルでは、大きなデータを一度に送ることができません。OS（Linux/macOS/Windows）のUSBドライバが自動的にデータを分割します：

```
例: 64KB のペイロードを送信する場合

OS が自動で分割:
  チャンク1: [16KB] → USB転送 →  SOV38受信
  チャンク2: [16KB] → USB転送 →  SOV38受信
  チャンク3: [16KB] → USB転送 →  SOV38受信
  チャンク4: [16KB] → USB転送 →  SOV38受信

このチャンクサイズが xperable の -b オプションで制御できる
```

**問題**: チャンクサイズによって、オーバーフローの位置やタイミングが変わります。

- チャンクが大きすぎる → オーバーフローが一気に起きて不安定
- チャンクが小さすぎる → 転送回数が増えてタイムアウトの可能性
- **ちょうどいいサイズ** → 環境（USBコントローラ、ドライバ、ケーブル）で異なる

### 自動最適化の動作

```
Phase 1: デフォルトサイズで 20回リトライ
  [████████████░░░░░░░░] #15 サイズ=デフォルト 試行 15/20
  → 5回ごとにUSB再接続を案内

Phase 2: サイズ変更して再試行 (20回 × 最大10サイズ)
  ┌────────────────────────┐
  │ デフォルト  → 20回試行  │ ✗
  │ 16KB (16384) → 20回試行 │ ✗
  │ 8KB (8192)   → 20回試行 │ ✓ 成功！(試行 #47)
  └────────────────────────┘

→ 最適バッファサイズ: 8192 bytes
→ 成功までの試行回数: 47
```

試行するバッファサイズの順序（SDM845向け）：

| 順番 | サイズ | 説明 |
|------|--------|------|
| 1 | デフォルト | xperable内蔵値 |
| 2 | 16,384 (16KB) | 一般的なUSBバルク転送サイズ |
| 3 | 8,192 (8KB) | 小さめ・安定性重視 |
| 4 | 32,768 (32KB) | 大きめ・高速 |
| 5 | 4,096 (4KB) | 最小・最安定 |
| 6 | 65,536 (64KB) | 最大 |
| 7-10 | 2KB, 48KB, 24KB, 12KB | 追加候補 |

### ログとレポート

すべての試行結果がJSONログに保存されます：

```
sov38_backups/
  exploit_logs/
    exploit_log_20260302_143022.json
```

全試行が失敗した場合、エラー内訳レポートが表示されます：

```
  エクスプロイト結果レポート
  総試行回数: 200
  試したバッファサイズ: デフォルト, 16384 bytes, 8192 bytes, ...

  エラー内訳:
    usb: 150回
    timeout: 30回
    unknown: 20回
```

### 成功率を上げるコツ

1. **短いUSBケーブル** — 長いケーブルはノイズが増える
2. **USB 2.0ポートを使う** — USB 3.0ハブ経由だと不安定になることがある
3. **直接接続** — USBハブを経由しない
4. **PC再起動** — USBドライバの状態をリセット
5. **別のPC** — USBコントローラの相性がある

## TAバックアップ機能

ヘルパーは以下の方法でTAバックアップを取得します：

1. **adb + root権限** — `dd` でTAパーティションを直接ダンプ
2. **xperable自動バックアップ** — xperable実行時に `TA.img` を自動作成

バックアップファイルは `sov38_backups/` に保存されます：

```
sov38_backups/
  TA_backup_20260302_120000.img      # adb経由バックアップ
  TA_xperable_20260302_130000.img    # xperable自動バックアップ
  boot_a_backup_20260302_120000.img  # boot imageバックアップ
```

:::danger バックアップは絶対に消さないで！
TAパーティションにはデバイス固有の暗号キーが含まれています。バックアップがないと復旧できない場合があります。必ず複数の場所にコピーしてください。
:::
